# SD1.5 Text-to-Image Design

## Goal

Add first-class text-to-image generation to cc-local using a local SD1.5-style checkpoint. The agent should be able to generate a PNG from a prompt, save it under the active workspace, and return a local file path that works in CLI, REPL, and Web chat.

## Decisions

- First version supports txt2img only.
- Use a persistent local Python T2I worker/server, not per-call model loading.
- Use `T2I/majicmixRealistic_v7.safetensors` as the default model.
- Do not require `T2I/v1-5-pruned-emaonly.safetensors` unless model switching is added later.
- Save outputs under the active runtime workspace `generated-images/` directory.
- Default image size is `512x768`, with explicit `512x512` and `768x512` still allowed.
- `generate_image` does not write Markdown files and does not create Markdown snippets as a primary output.
- Web Markdown preview should render image links such as `![image](generated-images/example.png)`.

## Reference Patterns

Hosted multimodal products generally treat generated images as model artifacts:

- OpenAI exposes image generation as a Responses API image generation tool, letting the model request image output as part of an agentic response flow.
- Gemini image generation uses multimodal content parts, where a response can include text and image data, and Imagen is exposed as a dedicated image generation model family.

This project already uses OpenAI-compatible chat completions plus local tools. The closest local equivalent is a normal tool call that returns image artifacts on disk.

References:

- https://platform.openai.com/docs/guides/tools-image-generation
- https://ai.google.dev/gemini-api/docs/image-generation

## Architecture

The main LLM remains the planner and tool caller. It does not load or run Stable Diffusion.

Add a TypeScript `generate_image` tool to the existing tool registry. The tool validates user-facing parameters, resolves the active runtime `cwd` output directory, calls a local Python T2I worker over HTTP, and returns a text result containing output paths and generation metadata.

Add a Python worker that starts separately from the agent runtime. It loads the SD1.5 checkpoint once at startup, exposes a small local API, and serializes generation requests so local GPU or unified memory is not overwhelmed.

## Data Flow

1. User asks the agent to generate an image.
2. Main LLM calls `generate_image`.
3. The tool validates parameters and computes the output directory from the current tool context `config.cwd`.
4. The tool sends `prompt`, generation parameters, and output directory to the T2I worker.
5. The worker generates one or more PNG files.
6. The worker returns paths, seeds, dimensions, and timing metadata.
7. The tool returns a readable result with absolute paths for CLI/REPL and workspace-relative paths for Markdown use.

## Worker API

The first version only needs:

- `GET /health`
- `POST /generate`

`POST /generate` accepts:

```json
{
  "prompt": "portrait photo ...",
  "negative_prompt": "low quality ...",
  "width": 512,
  "height": 768,
  "steps": 30,
  "cfg_scale": 7,
  "seed": 123,
  "count": 1,
  "output_dir": "/absolute/workspace/generated-images"
}
```

The worker returns:

```json
{
  "model": "majicmixRealistic_v7",
  "images": [
    {
      "path": "/absolute/workspace/generated-images/20260521-abc123.png",
      "seed": 123,
      "width": 512,
      "height": 768
    }
  ]
}
```

The worker should execute one generation request at a time. A small FIFO queue of size 1 or 2 is acceptable. If the queue is full, return a clear busy error.

## Tool Interface

Tool name: `generate_image`

Inputs:

```ts
{
  prompt: string
  negative_prompt?: string
  width?: number
  height?: number
  steps?: number
  cfg_scale?: number
  seed?: number
  count?: number
}
```

Defaults:

- `width`: `512`
- `height`: `768`
- `steps`: `30`
- `cfg_scale`: `7`
- `count`: `1`
- `seed`: random when omitted

Validation:

- `prompt` must be non-empty.
- `width` and `height` must be multiples of 64.
- `width * height` must not exceed `1024 * 1024`.
- `steps` must be within `1..60`.
- `cfg_scale` must be within `1..20`.
- `count` must be within `1..4`.

Tool output should be text-first so CLI/REPL works without special rendering:

```txt
Generated 1 image with majicmixRealistic_v7.
Image 1:
  absolute path: /Users/.../workspace/generated-images/20260521-abc123.png
  relative path: generated-images/20260521-abc123.png
  seed: 123
  size: 512x768
```

## Output Location

Outputs are scoped to the active runtime workspace, not `T2I/`.

For CLI/REPL, this means `config.cwd/generated-images/`. The user controls `config.cwd` through the process working directory or `--cwd`.

For Web runs, the server already rebuilds the agent runtime with the selected workspace path, so `config.cwd` is the active Web workspace.

Default workspace:

```txt
/Users/phoenix/Assistant/CC_based_Agent/workspace/generated-images/
```

Child workspace:

```txt
/Users/phoenix/Assistant/CC_based_Agent/workspace/<workspace-id>/generated-images/
```

The tool should create the directory when needed. `generated-images/` should be ignored by git.

## Web Markdown Preview

The Web UI should render images in Markdown preview when the selected `.md` file contains a relative image link:

```md
![image](generated-images/example.png)
```

Required changes:

- Extend the Markdown renderer to parse image syntax.
- Add a read-only workspace asset route for images.
- Resolve image paths against the active workspace.
- Reject path traversal, absolute paths, and files outside the active workspace.
- Serve only safe image types needed for generated output, initially PNG.

The generation tool does not automatically modify Markdown files. If the user asks the agent to add a generated image to a Markdown file, that should happen through the normal file tools after generation.

## Configuration

Add environment-driven configuration:

```env
T2I_BASE_URL=http://127.0.0.1:7861
T2I_MODEL_PATH=./T2I/majicmixRealistic_v7.safetensors
T2I_OUTPUT_DIR=generated-images
```

The TypeScript tool uses `T2I_BASE_URL` and `T2I_OUTPUT_DIR`. The Python worker uses `T2I_MODEL_PATH`. The model path should be absolute or resolved from the process root used to start the worker. The tool should pass the fully resolved output directory per request so the worker does not need to know workspace selection rules.

## Non-Goals

- No img2img.
- No inpainting.
- No Hires fix.
- No ADetailer, BMAB, or Dynamic Thresholding integration.
- No model switching UI.
- No automatic Markdown editing.
- No committing model weights or generated images.

## Verification

Implementation is complete when:

1. CLI can call the agent for image generation and the final response includes an existing `.png` absolute path.
2. REPL can do the same across more than one prompt.
3. Web chat can call the same tool and return the same path information.
4. Generated files are written under the active workspace `generated-images/` directory.
5. Web Markdown preview renders `![x](generated-images/file.png)` from the active workspace.
6. Invalid dimensions, excessive pixel count, invalid count, and unavailable worker produce clear tool errors.
7. Existing tests pass, and focused tests cover tool validation and Markdown image rendering.
